projects: MusiCube
Northwestern University, EECS 397: Digital Luthier, Professor Bryan Pardo - 2018
The UI controller is shown on the left of the image, next to the live MusiCube itself. The user can adjust musical and physical parameters using various buttons and sliders and then see them rendered live as the musical balls bounce around.

Overview
MusiCube is a Max/MSP application that combines audio and visual processing to create an intriguing musical experience for any user. The core of the application is a 3D cube that contains multiple naturally bouncing balls which sound different musical notes when they collide with eachother and strike different "wall" surfaces around the cube. I have also included an interface for users to control parameters such as the pitches of the collisions, number and size of balls, and physical attributes of the balls (restitution/"bounce"), among other attributes. Check out the demo at the bottom!

Technical Skills Used
  • Max/MSP, Jitter (jit.phys specifically)

Motivation
After spending some time previously working on a 2D "musical ball" square in C++/openFrameworks, I was interested in expanding this idea to the 3D space. Additionally, I was wanted to broaden my Max/MSP skill set (previously only Max for audio) to include Jitter, Max/MSP's library for graphics and visual processing. This is essentially a fun, visual toy for any user interested in combining and exploring audio and graphics. MusiCube could also be used as an inspirational tool for both electronic and acoustic composers alike.

Design
MusiCube is built fully using Cycling '74's Max/MSP and Jitter. The jit.phys package allows for naturally moving objects (i.e. realstic physical qualities of bouncing balls), so different elements of jit.phys were the core of the graphics portion of the code. Additionally, jit.phys allows for straightforward collision detection, which I used to dynamically detect when each ball collides with another surface. I then simply mapped these collisions to different MIDI note pitches such that they would play when each collision occured.

Musical Interaction/UI
Musical notes play whenever a ball strikes a wall of the cube or collides with another ball. Each wall is mapped to exactly one pitch, such that any ball that hits that specific wall plays that pitch. There is currently one universal pitch for any collisions between balls. Users can interact with MusiCube in many ways, both inside the actual cube and in the UI controller view. The external UI controller allows the user to change the number, color, and bounce level of balls, as well as the pitches of the walls of the cube, direction and force of gravity within the cube. The location of the partitions (shelves) within the cube can also be changed in this interface. Within the cube itself, users can drag and drop (or throw!) balls around the cube. The camera view of the cube can also be changed with simple mouse interaction.

Takeaways/Future Plans
I learned a lot about using Jitter in Max during this project which was very exciting and inspiring for future audio-visual projects. Max continues to impress me with its immense capability to do nearly anything from music to graphics to mathematics. Additionally, from this course I learned how to create an intuitive interface and target a specific user base, even though this specific project targets a wide user base and is more of a musical toy than a true instrument.

In the future, I would like to work to make the MusiCube more "musical", by connecting more physical components of the balls to musical elements such as dynamics and velocity of the note played. I would also like to implement more preset pitch patterns for the cube walls or the ability to easily change keys or move up or a down a key intuitively.
